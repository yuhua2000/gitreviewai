# 第三阶段需求 — 审计规则与配置管理

## 一、配置管理

### 1.1 业务配置迁移到 Web 端

将以下配置项从 `config.yaml` 迁移到数据库 + 前端管理界面：

| 配置项 | 说明 | 备注 |
|--------|------|------|
| `ignore_paths` | 忽略路径列表 | 迁移后支持 glob 模式（修复当前仅前缀匹配的 bug） |
| `max_line_comments` | 最大行评论数 | |
| `openai_model` | 默认 AI 模型 | 迁移为"全局默认模型"，项目可覆盖 |
| `openai_base_url` | AI API 基础地址 | |
| `openai_api_key` | AI API 密钥 | 敏感字段，前端需脱敏展示 |
| `gitlab_url` | GitLab 服务器地址 | 前端脱敏展示 |
| `gitlab_token` | GitLab Personal Access Token | 敏感字段，前端脱敏展示 |
| `webhook_token` | Webhook 验证 Token | 敏感字段，前端脱敏展示 |
| `log_level` | 日志级别 | 下拉选择 debug/info/warn/error |
| `jwt_expiry` | JWT 过期时间 | 如 24h、7d |

**保留在 config.yaml（基础设施配置，不迁移）：**
- `port`、`db_path`
- `password`、`jwt_secret`
- `encrypt_key`（安全敏感，不可在线修改）

**说明：** config.yaml 中的业务配置作为首次启动的初始值导入数据库，之后以数据库（Web 端）为准。

### 1.2 多模型管理

- 支持注册多个 AI 模型配置（名称、base_url、api_key、model_name）
- 每个模型配置独立管理，可启用/禁用
- 全局设置一个"默认模型"（用于未指定模型的项目）
- 前端提供模型列表管理页面

### 1.3 配置热更新

- Web 端修改配置后立即生效，无需重启服务
- 运行中的审查任务使用启动时的配置快照，新配置对下一次审查生效

## 二、审计规则管理

### 2.1 内置规则预设

提供以下预设规则分类，每条规则可单独启用/禁用：

#### ERROR 级别（默认全部启用）
| 规则 ID | 规则名称 | 说明 |
|---------|----------|------|
| `logic-error` | 逻辑错误 | 条件判断错误、循环边界、空值处理 |
| `security` | 安全漏洞 | SQL 注入、XSS、敏感信息泄露、权限绕过 |
| `concurrency` | 并发问题 | 竞态条件、死锁、资源未释放 |
| `error-handling` | 错误处理 | 异常吞没、错误未传递、资源泄漏 |
| `data-consistency` | 数据一致性 | 缺少事务、状态不同步 |

#### WARNING 级别（默认全部启用）
| 规则 ID | 规则名称 | 说明 |
|---------|----------|------|
| `performance` | 性能问题 | 不必要的循环、内存分配、N+1 查询 |
| `code-standards` | 代码规范 | 命名不清、魔法数字、重复代码 |
| `maintainability` | 可维护性 | 函数过长、嵌套过深、缺少注释 |
| `code-format` | 代码格式 | 缩进不一致、尾随空格、缺少空行、import 排序 |

#### INFO 级别（默认禁用）
| 规则 ID | 规则名称 | 说明 |
|---------|----------|------|
| `best-practice` | 最佳实践 | 语言惯用写法、社区最佳实践、设计模式建议 |
| `documentation` | 文档建议 | 缺少函数/模块注释、API 文档不完整 |

### 2.2 自定义规则

- 支持添加自定义规则条目（规则名称 + 规则描述文本）
- 自定义规则文本将注入到 AI 系统提示词中
- 自定义规则可设严重性级别（ERROR / WARNING / INFO）
- 可启用/禁用、编辑、删除

### 2.3 规则与项目绑定

- 每个仓库/项目可独立配置：
  - 启用哪些内置规则（默认继承全局配置）
  - 添加项目专属自定义规则
  - 绑定 AI 模型（默认继承全局默认模型）
  - 提交策略（auto_submit、触发条件）
- 全局默认配置 + 项目级覆盖的继承模式

### 2.4 项目自动发现

采用**自动建仓**方案：首次收到某项目的 MR webhook 时，自动创建项目配置记录，使用全局默认设置直接开始审查。

- 触发条件：webhook 收到 MR 事件，但 `project_configs` 表中无对应记录
- 自动创建行为：
  - 创建 `project_configs` 记录，`gitlab_project_id` 从 webhook 事件中提取
  - `project_name` 从 webhook 的 `project.name` 字段缓存
  - `description` 从 webhook 的 `project.description` 字段缓存
  - `ai_model_id` 为 NULL（继承全局默认模型）
  - 后续所有配置项使用全局默认值
  - `enabled = true`
- 创建后立即使用默认配置执行审查
- 用户事后可在前端项目配置页调整模型、规则、策略等

## 四、数据模型设计

### 4.1 新增表

```sql
-- AI 模型配置表
CREATE TABLE ai_models (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,          -- 显示名称，如 "GPT-4o"
    base_url    TEXT NOT NULL,                 -- API 地址
    api_key     TEXT NOT NULL,                 -- 加密存储
    model_name  TEXT NOT NULL,                 -- 模型标识，如 "gpt-4o"
    is_default  BOOLEAN NOT NULL DEFAULT 0,    -- 是否为全局默认模型
    enabled     BOOLEAN NOT NULL DEFAULT 1,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 审计规则表
CREATE TABLE review_rules (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_id     TEXT NOT NULL UNIQUE,          -- 规则标识，如 "logic-error"
    name        TEXT NOT NULL,                 -- 显示名称
    description TEXT,                          -- 规则描述
    severity    TEXT NOT NULL DEFAULT 'warning', -- error/warning/info
    is_builtin  BOOLEAN NOT NULL DEFAULT 1,    -- 是否内置规则
    enabled     BOOLEAN NOT NULL DEFAULT 1,    -- 全局是否启用
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 项目配置表（每个仓库的个性化配置）
CREATE TABLE project_configs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    gitlab_project_id INTEGER NOT NULL UNIQUE, -- GitLab 项目 ID
    project_name    TEXT,                      -- 项目显示名称（缓存）
    description     TEXT,                      -- 项目简介/描述
    ai_model_id     INTEGER,                   -- 绑定的 AI 模型 ID（NULL=继承默认）
    auto_submit     BOOLEAN DEFAULT 0,         -- 提交策略
    skip_draft      BOOLEAN DEFAULT 1,         -- 跳过 Draft MR
    target_branches TEXT,                      -- JSON 数组，目标分支列表
    ignore_paths    TEXT,                      -- JSON 数组，项目级忽略路径
    max_line_comments INTEGER,                 -- 项目级最大行评论数（NULL=继承默认）
    custom_prompt   TEXT,                      -- 自定义提示词（注入系统提示词）
    enabled         BOOLEAN NOT NULL DEFAULT 1,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 项目规则覆盖表（项目启用/禁用特定规则）
CREATE TABLE project_rule_overrides (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    project_config_id INTEGER NOT NULL,
    rule_id         TEXT NOT NULL,              -- 关联 review_rules.rule_id
    enabled         BOOLEAN NOT NULL,           -- 该项目下该规则是否启用
    UNIQUE(project_config_id, rule_id),
    FOREIGN KEY (project_config_id) REFERENCES project_configs(id)
);
```

### 4.2 现有表调整

- `settings` 表保留，用于存储全局业务配置（gitlab_url、gitlab_token、webhook_token、ignore_paths、max_line_comments、log_level、jwt_expiry 等迁移过来的配置项）
- 全局配置以 key-value 形式存储在 settings 表，与项目级配置形成继承关系
- 敏感字段（gitlab_token、openai_api_key、webhook_token）前端展示脱敏
- config.yaml 中的业务配置首次启动时作为默认值，Web 端修改后以数据库为准
- `log_level` 启动时从数据库读取，`jwt_expiry` 每次登录时从数据库读取

## 六、前端页面

### 6.1 新增页面

1. **AI 模型管理页** (`/settings/models`)
   - 模型列表（名称、base_url、model_name、默认标识、启用状态）
   - 新增/编辑弹窗（api_key 脱敏输入）
   - 设为默认、启用/禁用操作

2. **审计规则管理页** (`/settings/rules`)

3. **项目配置页** (`/projects` 及 `/projects/:id`)
   - 项目列表（项目名、绑定模型、规则覆盖数、启用状态）

     - 基本设置（绑定模型、auto_submit、skip_draft、target_branches）
     - 忽略路径配置（项目级，支持 glob）
     - 规则覆盖（表格形式，列出所有规则，项目可覆盖启用/禁用）
     - 自定义提示词编辑

### 6.2 现有页面调整

## 七、审查流程改造

### 7.1 系统提示词生成

审查时动态拼接系统提示词：

```
[基础系统提示词]
+
[启用的内置规则列表，含严重性和描述]
+
[自定义规则文本]
+
[项目自定义提示词]
```

### 7.2 审查触发流程改造

```
Webhook 收到 MR 事件
  → 从数据库获取全局配置（gitlab_url、gitlab_token 等）
  → 获取项目配置（project_configs）
     → 无记录？自动创建（使用全局默认），然后继续
     → 有记录但 enabled=false？跳过审查
  → 检查触发条件（skip_draft、target_branches）
  → 获取绑定的 AI 模型配置（base_url、api_key、model_name）
  → 动态生成系统提示词（合并规则）
  → 使用指定模型执行审查
  → 按项目配置的提交策略处理结果
```

## 八、实施优先级建议

1. **P0 — 基础设施**：数据模型 + settings 表扩展 + 配置迁移
2. **P1 — AI 模型管理**：模型 CRUD + 前端页面 + 审查流程接入
3. **P2 — 审计规则管理**：规则表 + 内置规则初始化 + 规则 CRUD + 前端页面
4. **P3 — 项目配置**：项目配置 CRUD + 规则覆盖 + 前端页面
5. **P4 — 流程集成**：动态提示词生成 + 触发条件 + 完整联调
